Guide de Soutenance Technique

Focus sur la partie Driver (Chauffeur) - SmartPickup

1. Architecture et Flux Global

La partie Driver gère l'état du chauffeur, sa localisation, et le cycle de vie des courses. Le backend est développé avec FastAPI et le frontend mobile avec Flutter. La communication se fait via des requêtes HTTP REST et des WebSockets pour le temps réel (notamment pour le suivi de la position).

2. Toutes les Fonctions du Driver (Backend)

Ces fonctions sont définies dans driver_controller.py et gèrent toute la logique métier du chauffeur.

GET get_profile

Route:
/profile/{user_id}
Description:
Récupère les informations du profil du chauffeur (nom, évaluation, etc.).

PUT update_profile

Route:
/profile/{user_id}
Description:
Met à jour les informations du profil.

PUT update_driver_status

Route:
/status/{user_id}
Description:
Bascule le statut du chauffeur entre En ligne (disponible) et Hors ligne.

PUT update_driver_location

Route:
/location/{user_id}
Description:
Met à jour les coordonnées GPS du chauffeur en base de données.

GET get_available_rides

Route:
/rides/available
Description:
Liste les courses en attente. Utilise en interne la recherche de proximité.

POST accept_ride

Route:
/rides/{request_id}/accept
Description:
Le chauffeur accepte une demande de course spécifique.

GET get_accepted_rides

Route:
/rides/accepted/{user_id}
Description:
Récupère la liste des courses acceptées par le chauffeur.

GET get_active_ride

Route:
/rides/active/{user_id}
Description:
Récupère la course actuellement en cours pour ce chauffeur.

GET get_driver_history

Route:
/rides/history/{user_id}
Description:
Récupère l'historique des courses terminées par le chauffeur.

PUT update_ride_status

Route:
/rides/{request_id}/status
Description:
Fait avancer le statut de la course (ex: 'arrived', 'started', 'completed').

GET get_taxi_info

Route:
/taxi/{user_id}
Description:
Récupère les informations sur le véhicule du chauffeur.

Interne _calculate_distance

Type:
Fonction d'aide (Helper)
Description:
Calcule la distance entre deux points GPS (formule de Haversine).

Interne _get_top_10_closest_drivers

Type:
Algorithme de recherche
Description:
Recherche et trie les 10 chauffeurs les plus proches d'une position donnée.

Interne _notify_ride_acceptance

Type:
Notification
Description:
Envoie une notification (via FCM ou WebSocket) pour confirmer l'acceptation de la course.

getProfile() Écran de Profil Affiche les informations personnelles et professionnelles du chauffeur.

updateLocation(lat, lng) Service GPS (Geolocator) Envoie la position toutes les X secondes. Permet le suivi en temps réel par le client.
getAvailableRides() Liste des courses disponibles Affiche les cartes des clients qui attendent un taxi.
acceptRide(requestId) Bouton "Accepter" Bascule l'application sur l'écran de navigation vers le client.
updateRideStatus(requestId, action) Boutons d'action (Arrivé, etc.) "Arrived" -> Notifie le client.
"Started" -> Lance le compteur/trajet.
"Completed" -> Termine et facture.
getActiveRide() Initialisation / Restauration Si l'application crash ou est fermée, cela permet de revenir directement à l'écran de la course en cours.

5. Résolutions et Logiques Clés (Questions du Jury)

Q: Comment fonctionne la mise à jour de la localisation ?

R: Le front utilise un plugin GPS (comme Geolocator) qui écoute les flux de position. Dès qu'un changement significatif est détecté, la méthode updateLocation est appelée pour synchroniser le backend.

Q: Comment gérez-vous la concurrence si deux chauffeurs acceptent la même course ?

R: Le backend (FastAPI) vérifie l'état de la course en base de données avant de valider l'acceptation. Si la course n'est plus "en attente" (pending), il renvoie une erreur au deuxième chauffeur, assurant que seul le premier a la course.

Q: Comment sécurisez-vous ces échanges ?

R: Tous les appels passent par des en-têtes HTTP incluant un Authorization: Bearer [JWT_TOKEN]. Le backend valide ce token pour identifier le chauffeur et vérifier ses droits.

Q: Pourquoi avoir choisi Mapbox plutôt que Google Maps ?

R: Mapbox offre une plus grande flexibilité pour la personnalisation des cartes (styles), une gestion performante des marqueurs personnalisés (comme l'icône du taxi qui tourne), et des API de directions très précises pour le calcul d'itinéraires.

6. Voyage d'une Donnée : Du Clic au Serveur

Voici le chemin précis parcouru par l'information lorsqu'un chauffeur clique sur un bouton (Exemple : "Accepter la course").

1
frontend/lib/screens/driver_home.dart Interaction UI : L'utilisateur clique sur le bouton. L'UI capture l'événement et récupère l'ID de la course.
2
frontend/lib/services/driver_api_service.dart Appel Service : La méthode acceptRide(id) est appelée. Elle construit la requête HTTP (URL, Headers avec Token JWT, Body).
3
backend/app/routes/driver_routes.py Point d'Entrée (Route) : Le serveur FastAPI reçoit la requête. La route @router.post("/{request_id}/accept") identifie l'action à effectuer.
4
backend/app/controllers/driver_controller.py Logique Métier : La fonction accept_ride() est exécutée. Elle vérifie si la course est toujours libre et si le chauffeur est éligible.
5
backend/app/database.py & models/ Persistance : Le contrôleur communique avec la base de données PostgreSQL pour mettre à jour l'état de la course (Passage de 'pending' à 'accepted').
6
Communication de Retour Réponse & UI Update : Le backend renvoie un code 200 (Success). Le frontend reçoit le JSON, met à jour l'état local, et navigue automatiquement vers l'écran de guidage.
